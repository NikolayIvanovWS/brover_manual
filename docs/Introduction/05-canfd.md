### CAN-интерфейсы на Raspberry Pi

На Raspberry Pi доступны два CAN-интерфейса:

- `can0` — используется системой ровера (связь драйверов моторов и бортового компьютера).

:::warning
В этот интерфейс не рекомендуется публиковать пользовательские сообщения, чтобы не нарушить работу платформы.
:::

- `can1` — пользовательский интерфейс.
Предназначен для подключения дополнительной полезной нагрузки (датчиков, модулей, cобственных контроллеров и т.д.).
Через него можно:
    - отправлять и принимать CAN-сообщения;
    - преобразовывать данные CAN в ROS-топики;
    - отправлять данные из ROS-топиков обратно в CAN.

### Параметры CAN-шины

На Raspberry Pi предустановлен пакет can-utils, интерфейсы запускаются автоматически.

Параметры шины:
- Тип: CAN FD
- Bitrate (арбитражная скорость): 1000000
- Dbitrate (скорость передачи данных): 8000000

Если требуется вручную запустить интерфейс can1:
```
sudo ip link set can1 up txqueuelen 65535 type can bitrate 1000000 dbitrate 8000000 fd on
```
Чтобы отключить интерфейс:
```
sudo ifconfig can1 down
```
### Основные утилиты can-utils
**Отправка сообщения**
```
cansend can1 <ID>#<data>
```
Мы используем расширенный 29-битный идентификатор (Extended ID) — это 8 hex-символов.

Пример: отправка сообщения с ID `0x12` и данными `DE AD BE EF`:
```
cansend can1 00000012#DEADBEEF
```

**Прослушивание шины**

Чтобы прослушать все сообщения на can1:
```
candump can1
```
Если нужно фильтровать по конкретному ID:
```
candump can1,<ID>:7ff
```
Например, чтобы слушать сообщения с ID `0x12`:
```
candump can1,0x12:7ff
```

**Проверка состояния интерфейса**
```
ip -details link show can1
```
Команда выводит подробную информацию об интерфейсе, включая:
- состояние интерфейса;
- режим (FD включен или нет);
- счётчики ошибок.

Если при передаче и приёме сообщений ошибок нет, соответствующие поля счётчиков будут равны 0.

### Важно: как правильно тестировать CAN

:::warning
 Некорректно тестировать CAN-шину, отправляя и принимая сообщения через один и тот же интерфейс (например, can1) на одной Raspberry Pi.
:::
Пример **некорректного** теста:

В одном терминале:
```
cansend can1 00000012#DEADBEEF
```
В другом терминале:
```
candump can1,0x12:7ff
```
Почему это неправильно?

Когда вы отправляете сообщение через `can1`, контроллер CAN может отразить его обратно в драйвер (так называемый *local loopback* или эхо переданного кадра). В этом случае вы видите сообщение в `candump`, даже если:
- на шине нет второго устройства;
- сообщение физически никто не принял;
- отсутствует подтверждение (ACK) от другого узла.

CAN-протокол предполагает, что кадр считается успешно переданным только при наличии подтверждения (ACK) от другого участника шины. Если на шине нет второго устройства, полноценной проверки передачи не происходит.

**Как тестировать правильно**

*Вариант 1 — **временно** соединить два интерфейса физически*

Соединить can0 и can1 проводами (CAN_H, CAN_L), и:

- отправлять сообщение в can1,
- принимать его в can0.

:::warning
Такое соединение допускается только для **кратковременного тестирования**.
После проверки работоспособности шины интерфейсы необходимо обязательно разъединить.
Интерфейс `can0` используется системой ровера для связи драйверов моторов и бортового компьютера. Постоянное подключение к нему пользовательских устройств или объединение его с `can1` может нарушить штатную работу платформы или привести к конфликтам сообщений
:::

*Вариант 2 — использовать второе устройство*

Подключить отдельный модуль контроллера (например, VBCore), прошитый для:

- приёма сообщений по CAN,
- или отправки сообщений.

Если сообщение действительно проходит от одного физического устройства к другому — шина работает корректно.
